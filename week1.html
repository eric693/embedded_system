嵌入式系統的定義、特性與分類
從根本問題出發，建立對嵌入式世界的準確認識
什麼是嵌入式系統？

嵌入式系統（Embedded System）是以特定應用為目標而設計的計算系統， 通常嵌入在更大型的機械或電子裝置之中，成為整體產品的一部分。它的核心目標不是提供通用計算能力， 而是在嚴格的資源與環境限制下，可靠地完成一組固定任務。

最簡單的判斷方式：如果一台電腦「被設計來做某幾件特定的事，而且通常不會被拿去做其他事」， 它就很可能是嵌入式系統。你的微波爐控制板、汽車的引擎控制單元（ECU）、門禁卡讀卡機， 都是嵌入式系統的典型例子。

常見誤解
「即時（Real-time）」並不等於「快速（Fast）」。 即時系統的核心要求是在確定且可預期的時間限制內完成任務。 一個 1 秒內必須完成的系統，和一個 1 毫秒內必須完成的系統，都可以是即時系統。 關鍵是「確定性（Determinism）」，而非絕對速度。
即時性分類：Hard RT vs. Soft RT vs. Firm RT

根據錯過 deadline 的後果嚴重程度，即時系統分為三類：

類別錯過 Deadline 的後果典型應用
Hard Real-Time 系統失敗，可能造成生命財產危險 汽車 ABS、飛機飛控、心律調整器
Firm Real-Time 任務結果作廢，不造成危害但品質下降 工業機器人、數位訊號處理
Soft Real-Time 效能降低，使用者體驗變差但可接受 視訊串流、遊戲渲染、UI 動畫

LESSON 1.2
應用領域與市場現況
了解嵌入式技術在各產業的滲透深度，以及各領域的技術要求差異
汽車電子（Automotive）

現代汽車是嵌入式系統密度最高的消費品之一。一輛高階電動車中，ECU（Electronic Control Unit）的數量可達 150 個以上，分別控制引擎、變速箱、ABS、氣囊、電動窗、車身穩定系統（ESC）、 ADAS 感測器融合等。這些 ECU 透過 CAN Bus、LIN Bus、FlexRay 和越來越普及的車載乙太網路（Automotive Ethernet）互相通訊。

汽車嵌入式開發最關鍵的規範是 ISO 26262（功能安全）和 AUTOSAR（軟體架構標準）。 前者要求開發過程中的每個環節都要有文件與驗證，後者定義了 BSW（基礎軟體）的分層架構， 使不同供應商的模組可以互通。

工業控制（Industrial Automation）

工廠自動化場景對確定性即時回應的要求極高。PLC（可程式邏輯控制器）、伺服驅動器、機器視覺系統 都是典型的工業嵌入式產品。工業環境的特殊挑戰在於電磁干擾（EMI）、寬溫工作範圍（-40°C 至 +85°C 或更高）、 以及長達 10~20 年的產品生命週期。

工業通訊協定以 Modbus、PROFIBUS、EtherCAT 為主流， 近年 TSN（Time-Sensitive Networking）正在推動工業以太網的實時化。

醫療設備（Medical Devices）

醫療嵌入式設備的核心要求是「絕對可靠」。一台輸液幫浦、呼吸機或除顫器，容不得任何軟體錯誤。 除了功能安全（IEC 62304 醫療軟體生命週期標準）之外， 資料安全（患者隱私、HIPAA）和電磁兼容性（EMC）也是強制要求。

物聯網與消費性電子（IoT & Consumer）

IoT 感測器節點的典型需求是：極低功耗（電池供電數年）、無線連接（BLE、Zigbee、LoRa、NB-IoT）、 以及安全的 OTA（Over-The-Air）韌體更新能力。消費性電子則更強調成本控制、尺寸最小化， 以及豐富的用戶體驗（觸控、語音、顯示）。

主流處理器架構全覽
了解 ARM、AVR、RISC-V 的架構差異與選型邏輯
指令集架構（ISA）的基本概念

指令集架構（Instruction Set Architecture, ISA）定義了處理器能理解的機器指令集合， 包括指令格式、暫存器數量與寬度、定址模式、以及記憶體模型。ISA 是硬體與軟體之間的契約介面。

現代嵌入式 ISA 主要分為兩大設計哲學： RISC（精簡指令集）— 指令數量少、格式固定、執行週期可預測，有利於編譯器最佳化； CISC（複雜指令集）— 單一指令能完成複雜操作，程式碼密度高，x86 是代表。 嵌入式領域幾乎由 RISC 架構主導。

ARM Cortex 系列

ARM Holdings 設計 ISA 但不自行製造晶片，而是授權給 ST、NXP、Nordic、TI 等廠商。 這種 IP 授權模式使 ARM 架構成為全球嵌入式市場的絕對主流。

ARM 的 Cortex 系列針對不同應用分為三個剖面（Profile）：

系列定位代表核心典型應用
Cortex-A 高性能應用處理器，支援 MMU，可跑完整 Linux A53, A72, A55 Raspberry Pi, 手機 SoC, 車載資訊娛樂
Cortex-R 高確定性即時，支援 MPU，用於安全關鍵 R5, R8 硬碟控制器、汽車 ABS、工業伺服
Cortex-M 微控制器專用，低功耗、低成本、易開發 M0, M0+, M3, M4, M7, M33 STM32、nRF52、RP2040 等絕大多數 MCU

本課程使用的 STM32F446RE 搭載 Cortex-M4F 核心， 「F」代表內建 FPU（浮點運算單元），主頻最高 180 MHz， 具有單精度浮點硬體加速，適合感測器訊號處理與馬達控制。

AVR 架構

Atmel（現為 Microchip）的 AVR 是一款 8 位元 RISC 架構，以「哈佛架構」設計聞名 ——程式記憶體（Flash）與資料記憶體（SRAM）分離，各有獨立匯流排，提升取指效率。 AVR 的指令集設計讓大多數指令在單一時脈週期完成，非常適合低功耗應用。

Arduino Uno 使用的 ATmega328P（16 MHz、32 KB Flash、2 KB RAM）是 最廣為人知的 AVR 成員，也是嵌入式入門的重要學習工具。 然而 8 位元架構在處理 32 位元資料時效率較低，且缺乏硬體浮點運算， 在效能需求較高的場合已逐漸被 ARM Cortex-M 取代。

RISC-V 開放架構

RISC-V 是由 UC Berkeley 於 2010 年發起的開放指令集架構， 任何人都可以免費實作、修改、甚至商用，無需支付授權費。 這使得 RISC-V 在 IoT、邊緣 AI 和開源硬體社群中快速崛起。

RISC-V 採用模組化設計：基礎整數指令集（RV32I / RV64I）加上可選擴充（M：乘除法、 A：原子操作、F/D：浮點、C：壓縮指令），靈活度極高。 代表性 MCU 包括 GigaDevice 的 GD32VF103 以及近年爆紅的 ESP32-C3（內建 Wi-Fi + BLE 的 RISC-V SoC）。

本課程選擇 STM32F446RE（Cortex-M4F） 作為主要實作平台，原因如下： 社群資源豐富、HAL 庫完整、ST-Link 偵錯器普及、覆蓋了大量工業與消費性電子實際應用場景， 且 Cortex-M4F 的 DSP 指令與 FPU 使後續的訊號處理章節可以直接在硬體上實測。

嵌入式開發的完整流程

連結腳本（Linker Script）基本概念

連結腳本（.ld 檔案）告訴連結器如何將各個目的檔的段（Section） 對應到目標晶片的物理記憶體位址。這是嵌入式開發中最獨特也最重要的概念之一。

linker script — STM32F446RE.ld（簡化版） 
/* 定義記憶體區域：Flash 起始 0x08000000，大小 512 KB
                        RAM  起始 0x20000000，大小 128 KB */
MEMORY
{
  FLASH  (rx)  : ORIGIN = 0x08000000, LENGTH = 512K
  RAM    (rwx) : ORIGIN = 0x20000000, LENGTH = 128K
}

SECTIONS
{
  /* .text 段：可執行程式碼，放進 Flash */
  .text :
  {
    KEEP(*(.isr_vector))   /* 中斷向量表必須放在最前面 */
    *(.text*)
    *(.rodata*)            /* 唯讀常數也放 Flash */
  } > FLASH

  /* .data 段：已初始化全域變數，執行時複製到 RAM */
  .data :
  {
    _sdata = .;
    *(.data*)
    _edata = .;
  } > RAM AT> FLASH       /* 載入位址在 Flash，執行位址在 RAM */

  /* .bss 段：未初始化全域變數，RAM 中清零 */
  .bss :
  {
    _sbss = .;
    *(.bss*)
    *(COMMON)
    _ebss = .;
  } > RAM
}
重要觀念
_sdata、_edata、_sbss、_ebss 這些符號由連結腳本定義， 在啟動碼（startup code）中使用，負責將 .data 從 Flash 複製到 RAM、 以及將 .bss 段清零。這是嵌入式程式執行 main() 之前必須完成的初始化工作。
建置系統：Makefile vs. CMake

小型專案通常使用 Makefile，直接描述編譯規則。 中大型專案或需要跨平台建置的專案則使用 CMake， 透過 CMakeLists.txt 生成各平台的建置檔。 STM32CubeIDE 和 VS Code + Cortex-Debug 擴充套件都支援 CMake 工作流。

實作範例
LED Blink — 用 HAL 庫實作（初學版）

STM32F446RE Nucleo 開發板的板載 LED 連接在 PA5（GPIO Port A, Pin 5）。 以下使用 STM32 HAL 庫撰寫最簡潔的 Blink 程式：

c — main.c（HAL 版 LED Blink） 
#include "stm32f4xx_hal.h"

void SystemClock_Config(void);

int main(void)
{
    /* 初始化 HAL 庫（必須最先呼叫）*/
    HAL_Init();

    /* 設定系統時脈（使用外部晶振 HSE，PLL 倍頻至 180 MHz）*/
    SystemClock_Config();

    /* 啟用 GPIOA 的時脈（預設關閉以省電）*/
    __HAL_RCC_GPIOA_CLK_ENABLE();

    /* 設定 PA5 為推挽輸出，無上下拉，低速模式 */
    GPIO_InitTypeDef gpio = {0};
    gpio.Pin   = GPIO_PIN_5;
    gpio.Mode  = GPIO_MODE_OUTPUT_PP;
    gpio.Pull  = GPIO_NOPULL;
    gpio.Speed = GPIO_SPEED_FREQ_LOW;
    HAL_GPIO_Init(GPIOA, &gpio);

    while (1)
    {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
        HAL_Delay(500);   /* 500 ms，由 SysTick 計時 */
    }
}
LED Blink — 直接操作暫存器（進階版）

HAL 庫封裝了大量細節，但作為工程師必須理解底層行為。 以下是不依賴任何庫、直接操作記憶體映射暫存器（Memory-Mapped Register）的版本：

c — main.c（暫存器直接操作版）
#include <stdint.h> /* RCC 與 GPIOA 基底位址（來自 STM32F4xx Reference Manual）*/ #define RCC_BASE 0x40023800U #define GPIOA_BASE 0x40020000U /* RCC AHB1 周邊時脈致能暫存器（偏移 0x30）*/ #define RCC_AHB1ENR (*((volatile uint32_t *)(RCC_BASE + 0x30))) /* GPIOA 暫存器（各偏移量見 RM0390 第 8 章）*/ #define GPIOA_MODER (*((volatile uint32_t *)(GPIOA_BASE + 0x00))) #define GPIOA_ODR (*((volatile uint32_t *)(GPIOA_BASE + 0x14))) #define GPIOA_BSRR (*((volatile uint32_t *)(GPIOA_BASE + 0x18))) static void delay_ms(uint32_t ms) { /* 粗略延遲：假設系統時脈為 16 MHz（內部 HSI）*/ for (uint32_t i = 0; i < ms * 2000; i++) __asm__("nop"); } int main(void) { /* 步驟 1：啟用 GPIOA 時脈（RCC_AHB1ENR bit 0）*/ RCC_AHB1ENR |= (1U << 0); /* 步驟 2：設定 PA5 為一般輸出（MODER[11:10] = 0b01）*/ GPIOA_MODER &= ~(3U << (5 * 2)); /* 先清除 PA5 的 MODER 位元 */ GPIOA_MODER |= (1U << (5 * 2)); /* 設為 01 = General purpose output */ while (1) { /* 使用 BSRR 進行原子性 Set（高 16 bit）/ Reset（低 16 bit）*/ GPIOA_BSRR = (1U << 5); /* Set PA5 = LED ON */ delay_ms(500); GPIOA_BSRR = (1U << (5 + 16)); /* Reset PA5 = LED OFF */ delay_ms(500); } }
為什麼用 BSRR 而不用 ODR？
直接讀改寫（Read-Modify-Write）ODR 在多中斷環境下有 race condition 風險： 若在「讀取」和「寫入」之間發生中斷並修改了同一個 port 的其他 pin， 就會產生意外的值被覆蓋。BSRR 是一個只寫、原子性的暫存器， Set 和 Reset 用不同的位元控制，不需要先讀取，因此是 GPIO 操作的推薦方式。
