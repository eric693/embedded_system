5.1 — 理論
RTOS 概念與 FreeRTOS 架構
為什麼需要 RTOS、決定性時序、FreeRTOS 核心組成
裸機 Super Loop vs. RTOS

裸機超迴圈（Super Loop）在功能單純時夠用，但當需求增加時， 各功能的最壞情況執行時間（WCET）互相疊加， 導致無法保證任何一個功能的回應時間。

Super Loop 問題： time → [UART 處理 ][感測器讀取 ][顯示更新 ][CAN 傳送 ] └──────────────────────────────────────────────────────┘ ← 若 CAN 傳送卡住，UART 就錯過資料 → RTOS 搶佔式排程： Tick IRQ 每 1 ms 強制切換高優先任務 [UART Hi-P][Sensor ][UART Hi-P][Display ][UART Hi-P] 每個任務最壞回應時間 = 所有更高優先任務的 WCET 之和（可計算、可保證）

組件原始檔功能

佇列queue.cQueue、Semaphore、Mutex 底層實作
計時器timers.c軟體計時器（需 Timer Daemon Task）
事件群組event_groups.c多 bit 事件旗標，支援等待任意組合
記憶體heap_1–5.c五種堆積配置策略
移植層port.c / portmacro.h架構相依：上下文切換、SysTick、PendSV
FreeRTOS 與 STM32 的整合方式

FreeRTOS 使用 SysTick 作為 Tick 計時器（預設 1 ms）， 使用 PendSV（最低優先權中斷）執行實際的上下文切換。 這樣設計確保上下文切換不會打斷真正的硬體中斷， 而是等所有硬體 ISR 都處理完後，再在 PendSV 中完成任務切換。

FreeRTOS 中斷優先權規則
在 FreeRTOSConfig.h 中必須設定：
configMAX_SYSCALL_INTERRUPT_PRIORITY（通常 = 5，即優先數值 ≥ 5 的 ISR 才能呼叫 FreeRTOS API）

呼叫 xQueueSendFromISR、xSemaphoreGiveFromISR 等 ...FromISR 函式的 ISR， 其 NVIC 優先數值必須 大於或等於 configMAX_SYSCALL_INTERRUPT_PRIORITY。 數值越大 = 優先權越低；數值 0–4 的高優先 ISR 不得呼叫任何 FreeRTOS API。
5.2 — 理論 + 實作
任務（Task）管理
任務狀態機、建立刪除、堆疊大小估算、vTaskDelay vs. vTaskDelayUntil
任務狀態機
xTaskCreate() │ ▼ [ Ready ] ◀────────────────────────┐ │ │ 排程器選中 │ vTaskDelay / 等待事件完成 ▼ │ [ Running ] ──── 等待資源/延遲 ──▶ [ Blocked ] │ 更高優先任務搶佔 │ ▼ [ Suspended ] ◀── vTaskSuspend() │ vTaskResume() / vTaskDelete()

任何時刻只有一個任務處於 Running 狀態（單核 MCU）。 Blocked 狀態的任務不佔用 CPU，等待條件滿足（計時到期、Queue 有資料、Semaphore 可取得）後 自動轉回 Ready。

建立任務與堆疊估算
c — 建立兩個任務：LED 閃爍 + UART 監控
#include "FreeRTOS.h" #include "task.h" /* 任務函式：必須永不返回（無限迴圈），或以 vTaskDelete(NULL) 終止 */ static void task_led(void *arg) { (void)arg; for (;;) { gpio_toggle(LED_GREEN_PORT, LED_GREEN_PIN); vTaskDelay(pdMS_TO_TICKS(500)); /* 讓出 CPU，500 ms 後回到 Ready */ } } static void task_monitor(void *arg) { (void)arg; for (;;) { /* 印出每個任務的堆疊剩餘空間（High Water Mark）*/ uart_printf("LED stack remaining: %u words\r\n", (unsigned)uxTaskGetStackHighWaterMark(led_handle)); vTaskDelay(pdMS_TO_TICKS(5000)); } } int main(void) { SystemClock_Config(); uart2_init(115200); /* xTaskCreate(函式, 名稱, 堆疊大小(words), 參數, 優先權, 控制碼) */ xTaskCreate(task_led, "LED", 128, NULL, 2, &led_handle); xTaskCreate(task_monitor, "Monitor", 256, NULL, 1, NULL); vTaskStartScheduler(); /* 啟動排程器，不應返回 */ for (;;); /* 到這裡代表記憶體不足 */ }
堆疊大小如何估算
堆疊大小單位是 word（4 bytes），而非 bytes。
1. 初期設大一點（512–1024 words）
2. 執行後呼叫 uxTaskGetStackHighWaterMark() 查剩餘最小值
3. 剩餘值 = 從未被使用過的最小 word 數，建議保留 ≥ 20% 餘量
4. 使用 configCHECK_FOR_STACK_OVERFLOW = 2 啟用溢位偵測 hook
vTaskDelay vs. vTaskDelayUntil
函式延遲起點週期誤差適用場景
vTaskDelay(n) 呼叫當下 會累積（執行時間 + 延遲） 非精確週期工作
vTaskDelayUntil(&last, n) 上次喚醒時間 固定週期，不受執行時間影響 精確週期：控制迴圈、感測器取樣
c — vTaskDelayUntil 精確 10 ms 控制週期
static void task_control_loop(void *arg) { (void)arg; TickType_t last_wake = xTaskGetTickCount(); const TickType_t period = pdMS_TO_TICKS(10); /* 10 ms = 100 Hz */ for (;;) { read_sensors(); run_pid_controller(); update_actuators(); /* 無論上方耗時多少，下次喚醒仍在 last_wake + period */ vTaskDelayUntil(&last_wake, period); } }
5.3 — 理論
排程器與優先權
搶佔式 vs. 合作式、優先權反轉、Rate Monotonic 分析
FreeRTOS 排程策略

FreeRTOS 預設使用固定優先權搶佔式（Fixed Priority Preemptive）排程。 優先權數值越大代表優先權越高（與 NVIC 相反）。 同優先權的多個 Ready 任務以時間片輪轉（Round-Robin）方式分配， 每個 Tick 切換一次（需啟用 configUSE_TIME_SLICING = 1）。

Rate Monotonic 優先權分配

Rate Monotonic（RM） 是最常用的靜態優先權分配準則： 執行週期越短的任務，優先權越高。RM 排程在 CPU 使用率 ≤ 69% 時 可保證所有任務皆符合截止期限（Deadline）。

任務週期WCET使用率RM 優先權
控制迴圈10 ms2 ms20%最高（3）
感測器讀取50 ms5 ms10%中（2）
UART 監控1000 ms10 ms1%最低（1）
總使用率31%✓ 安全（< 69%）
優先權反轉（Priority Inversion）

當低優先任務持有高優先任務所需的資源（Mutex）時， 若中優先任務搶佔低優先任務，高優先任務會被「間接阻塞」， 其實際優先權被反轉至比中優先任務更低，破壞即時性保證。

時間 → Task H（高） ──執行── 等 Mutex ─────────────────── 搶回 ──▶ Task M（中） ──── 搶佔 L ──── 執行直到完成 ────▶ Task L（低） ──持 Mutex── 被 M 搶佔 ──────────────── 釋放 ──▶ ▲ H 被 M 間接阻塞（Priority Inversion） 解法：Mutex Priority Inheritance（優先權繼承） → L 持有 Mutex 期間，暫時繼承 H 的優先權，M 無法搶佔 L → FreeRTOS Mutex（xSemaphoreCreateMutex）自動實作此機制
5.4 — 理論 + 實作
Queue 佇列
FIFO 複製語意、ISR 安全 API、Queue Set 與 Mailbox
Queue 核心概念

FreeRTOS Queue 採用複製語意（Copy by Value）： 傳送時將資料完整複製進佇列緩衝區，接收時再複製出來。 這避免了指標失效（dangling pointer）問題，但對大型資料應傳指標（本身）而非整個結構。 Queue 的所有 Block 操作皆為安全的，內部以臨界區保護。

ISR ↔ 任務資料傳遞
c — UART ISR 將收到的 byte 送入 Queue，任務消費
QueueHandle_t q_uart_rx; void app_init(void) { /* 建立深度 64、元素大小 1 byte 的佇列 */ q_uart_rx = xQueueCreate(64, sizeof(uint8_t)); configASSERT(q_uart_rx != NULL); } /* ISR：不能 Block，使用 FromISR 版本 */ void USART2_IRQHandler(void) { BaseType_t higher_prio_woken = pdFALSE; if (USART2->SR & USART_SR_RXNE) { uint8_t b = (uint8_t)USART2->DR; xQueueSendFromISR(q_uart_rx, &b, &higher_prio_woken); } /* 若喚醒了高優先任務，通知排程器在 ISR 退出後立即切換 */ portYIELD_FROM_ISR(higher_prio_woken); } /* 消費任務：Block 等待，0 CPU 浪費 */ static void task_uart_proc(void *arg) { (void)arg; uint8_t byte; for (;;) { if (xQueueReceive(q_uart_rx, &byte, pdMS_TO_TICKS(100)) == pdTRUE) { protocol_parse_byte(byte); } } }
ISR 中絕對不能使用無 FromISR 後綴的 API
在 ISR 中呼叫 xQueueSend（非 FromISR 版）會進入 Block 狀態， 而 ISR 的上下文不支援排程器，將導致 HardFault 或系統卡死。 所有在 ISR 中呼叫的 FreeRTOS 函式必須使用 ...FromISR 後綴版本。
Queue Set — 等待多個佇列之一

當一個任務需要回應多個資料來源（例如同時等待 UART、CAN、按鍵）， 可以將多個 Queue / Semaphore 加入 Queue Set， 以單一 xQueueSelectFromSet 阻塞等待任意一個有資料。 這比輪詢多個佇列更有效率，也避免使用全域旗標。

5.5 — 理論 + 實作
Semaphore 與 Mutex
Binary、Counting Semaphore、Mutex 與遞迴 Mutex 的正確使用場合
三種同步原語比較
類型建立初始值主要用途Priority Inheritance
Binary Semaphore xSemaphoreCreateBinary() 0（空） ISR → 任務 單次事件通知 否
Counting Semaphore xSemaphoreCreateCounting(max, init) 自定 資源池計數（緩衝區槽位） 否
Mutex xSemaphoreCreateMutex() 1（可取得） 共享資源互斥存取 有
Recursive Mutex xSemaphoreCreateRecursiveMutex() 1 同一任務可重複取得（可重入函式） 有
Binary Semaphore：ISR 事件通知
c — 按鍵 EXTI ISR → 任務（Binary Semaphore）
SemaphoreHandle_t sem_btn; void EXTI15_10_IRQHandler(void) { BaseType_t woken = pdFALSE; if (EXTI->PR & EXTI_PR_PR13) { EXTI->PR = EXTI_PR_PR13; xSemaphoreGiveFromISR(sem_btn, &woken); } portYIELD_FROM_ISR(woken); } static void task_button(void *arg) { (void)arg; sem_btn = xSemaphoreCreateBinary(); for (;;) { /* 永久等待，不輪詢，0 CPU 消耗 */ if (xSemaphoreTake(sem_btn, portMAX_DELAY) == pdTRUE) { gpio_toggle(LED_GREEN_PORT, LED_GREEN_PIN); } } }
Mutex：保護共享 UART 輸出
c — Mutex 保護多任務競爭的 UART printf
SemaphoreHandle_t mtx_uart; void uart_printf_safe(const char *fmt, ...) { if (xSemaphoreTake(mtx_uart, pdMS_TO_TICKS(100)) == pdTRUE) { char buf[128]; va_list args; va_start(args, fmt); vsnprintf(buf, sizeof(buf), fmt, args); va_end(args); uart_send_blocking((uint8_t *)buf, strlen(buf)); xSemaphoreGive(mtx_uart); } } /* 任何任務皆可安全呼叫，不會輸出亂碼 */ static void task_sensor(void *arg) { for (;;) { float temp = bme280_read_temp(); uart_printf_safe("Sensor: %.1f C\r\n", temp); vTaskDelay(pdMS_TO_TICKS(1000)); } }
Mutex 絕對不能在 ISR 中使用
Mutex 具有優先權繼承機制，其 Give 操作可能觸發任務切換， 而 ISR 不支援此行為。ISR 中同步應用 Binary Semaphore， 共享資源互斥應在任務層用 Mutex。
5.6 — 理論 + 實作
軟體計時器與事件群組
One-shot / Auto-reload Timer、Event Group 多 bit 等待、Task Notification
軟體計時器

FreeRTOS 軟體計時器由Timer Daemon Task（優先權可設定）管理。 回呼函式（Callback）在 Timer Daemon 的上下文中執行， 因此回呼函式不得進行 Block 操作，且執行時間應盡量短。

c — One-shot 計時器：按鍵消抖（50 ms 後確認）
TimerHandle_t tmr_debounce; static void debounce_cb(TimerHandle_t xTimer) { (void)xTimer; /* 50 ms 後再確認按鍵狀態 */ if (!gpio_read(BTN_PORT, BTN_PIN)) { /* 低 = 仍按著 */ xSemaphoreGive(sem_btn_confirmed); } } void EXTI15_10_IRQHandler(void) { BaseType_t woken = pdFALSE; if (EXTI->PR & EXTI_PR_PR13) { EXTI->PR = EXTI_PR_PR13; /* 重置計時器（若已在執行則重新計時，實現消抖）*/ xTimerResetFromISR(tmr_debounce, &woken); } portYIELD_FROM_ISR(woken); } void timers_init(void) { /* pdFALSE = One-shot；pdTRUE = Auto-reload */ tmr_debounce = xTimerCreate("Debounce", pdMS_TO_TICKS(50), pdFALSE, NULL, debounce_cb); configASSERT(tmr_debounce != NULL); }
事件群組（Event Group）

Event Group 以 24-bit 旗標（bit 0–23 可用）表示事件狀態， 支援等待任意 bit 組合（OR）或全部 bit 都設定（AND）， 適合需要等待「多個條件同時成立」的場景。

c — Event Group：等待感測器 + 網路同時就緒才啟動
#define EVT_SENSOR_READY (1 << 0) #define EVT_NETWORK_READY (1 << 1) #define EVT_ALL_READY (EVT_SENSOR_READY | EVT_NETWORK_READY) EventGroupHandle_t eg_system; static void task_sensor_init(void *arg) { (void)arg; bme280_init(); xEventGroupSetBits(eg_system, EVT_SENSOR_READY); vTaskDelete(NULL); /* 初始化完成，任務自我刪除 */ } static void task_main(void *arg) { (void)arg; /* 等待所有就緒位元（AND 等待），等待後自動清除旗標 */ xEventGroupWaitBits(eg_system, EVT_ALL_READY, /* 等待的 bits */ pdTRUE, /* 等到後清除 */ pdTRUE, /* AND 模式（全部都要設定）*/ portMAX_DELAY); /* 到這裡，感測器與網路都已就緒 */ start_main_application(); for (;;) { /* ... */ } }
Task Notification — 輕量替代 Semaphore

每個任務內建一個 32-bit 通知值，可直接作為輕量的 Binary Semaphore、 Counting Semaphore 或 Mailbox 使用，無需額外建立核心物件， 速度比 Queue/Semaphore 快約 45%，RAM 消耗更低。

c — Task Notification 取代 Binary Semaphore
static TaskHandle_t h_task_proc; /* ISR：通知特定任務 */ void USART2_IRQHandler(void) { BaseType_t woken = pdFALSE; if (USART2->SR & USART_SR_RXNE) { (void)USART2->DR; vTaskNotifyGiveFromISR(h_task_proc, &woken); } portYIELD_FROM_ISR(woken); } static void task_proc(void *arg) { (void)arg; h_task_proc = xTaskGetCurrentTaskHandle(); for (;;) { ulTaskNotifyTake(pdTRUE, portMAX_DELAY); /* 等待通知 */ process_incoming_data(); } }
5.7 — 理論 + 實作
記憶體管理
五種 heap 方案、靜態分配、記憶體使用監控與最佳實踐
FreeRTOS 五種記憶體配置方案
方案可釋放碎片化執行緒安全適用場景
heap_1否無是啟動時一次分配完畢，不刪除任務
heap_2是有（不合併）是固定大小的重複分配/釋放
heap_3是視 malloc需 Mutex包裝系統 malloc/free
heap_4是有（鄰近合併）是最常用：可混合大小分配
heap_5是有（鄰近合併）是多段非連續 RAM（SRAM1 + SRAM2）
嵌入式最佳實踐
優先使用 heap_4，並在系統啟動時完成所有 xTaskCreate、 xQueueCreate 等分配，之後不再呼叫 pvPortMalloc。 這樣等同於 heap_1 的無碎片化特性，同時保有 heap_4 的靈活性。
靜態分配（Static Allocation）

啟用 configSUPPORT_STATIC_ALLOCATION = 1 後， 可將任務堆疊與 TCB（Task Control Block）分配在靜態或全域陣列， 完全避免動態 heap 分配，RAM 用量完全可預測， 適合安全關鍵系統（IEC 61508、DO-178C）。

c — 靜態分配任務與佇列
/* 靜態任務所需的記憶體（編譯期確定大小）*/ static StaticTask_t led_tcb; static StackType_t led_stack[128]; /* 128 words = 512 bytes */ /* 靜態佇列所需的記憶體 */ static StaticQueue_t q_storage; static uint8_t q_buf[64]; /* 64 × sizeof(uint8_t) */ void app_create_tasks(void) { /* 傳入靜態儲存空間，不使用 heap */ xTaskCreateStatic(task_led, "LED", 128, NULL, 2, led_stack, &led_tcb); QueueHandle_t q = xQueueCreateStatic(64, sizeof(uint8_t), q_buf, &q_storage); (void)q; } /* 當 configSUPPORT_STATIC_ALLOCATION = 1 時， 必須提供這兩個 Idle Task 和 Timer Task 的靜態記憶體 hook */ void vApplicationGetIdleTaskMemory( StaticTask_t **ppTCB, StackType_t **ppStack, uint32_t *pSize) { static StaticTask_t idle_tcb; static StackType_t idle_stack[configMINIMAL_STACK_SIZE]; *ppTCB = &idle_tcb; *ppStack = idle_stack; *pSize = configMINIMAL_STACK_SIZE; }
